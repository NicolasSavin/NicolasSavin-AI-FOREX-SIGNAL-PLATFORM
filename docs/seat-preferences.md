# Требования к местам в железнодорожных вагонах

Система поддерживает объект `seat_preferences` в заявке поиска и сохранённом поиске. Он хранит классы вагона, предпочтение нижних/верхних мест, размещение в одном купе или вагоне, соседние места, исключение боковых мест, пол купе, максимум купе и режим `strict_preferences`.

`Одно купе` означает, что для пассажиров найдётся нужное количество подходящих свободных мест в одном `compartment_number`. Это не означает выкуп целого купе.

`Купе без посторонних` (`require_private_compartment`) требует доступности всех мест купе и соответствия правилу, даже если пассажиров меньше вместимости купе.

`SeatAllocationService` получает число пассажиров, `SeatPreferences` и список универсальных `RailwayPlace`. Он фильтрует недоступные места, классы вагона, боковые места, нижние/верхние места и гендерные ограничения, затем подбирает комбинацию по купе, вагону, соседству, разделению группы и `maximum_compartments`. Сервис ничего не бронирует и не удерживает.

Яндекс Расписания предоставляет расписание, но не подтверждённую схему конкретных мест. Поэтому при отсутствии схемы мест результат получает статус `unknown`, предупреждение «Источник расписания не предоставляет схему мест» и не становится `available` только по расписанию.

B2B-провайдер должен предоставить нормализуемые данные: номер поезда, номер вагона, тип вагона, сервисный класс, номер купе, номер места, тип/позицию места, боковое ли место, ограничение пола, доступность места и время получения данных. Полный сырой ответ внешнего API не сохраняется.
